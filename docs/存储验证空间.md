# 存储验证和用户空间

## 存储验证机制

预计的直链形式：https://example.com/raw/test/demo.txt&sign=xxx&crypt=xxx
也许可以换成authority
链接也可以直接包含创建时间和有效期，直接验证签名就行？

- `sign=xxx`为签名密文，需要解密验证是否有效
- `crypt=xxx`为指定密钥组UUID（或其密文），可根据后端密钥组进行解密

### 签名结构

- 明文由以下内容构成`有效时间` `路径短哈希（作为盐值）` `路径` `链接创建时间`
- 由用户系统使用主密钥/证书对明文进行加密，作为访问路径的权限证明。
- 文件系统对链接的签名进行解密，如果密码不对则解析不符合结构，返回错误。
- 文件系统缓存有效的签名，避免重复解密判断，TTL应当根据链接有效时间进行，如果为永久有效，应当有一个默认TTL时间。

### 前提

- 主密钥/证书应当被认为是安全的，只有用户系统/插件管理可以接触。
- 不会滥用永久有效链接。

## 用户空间

用户空间将真实的挂载结构和用户自身的路径结构分开

### 用户管理

- 默认为继承权限组权限？
- 预计的链接形式：https://example.com/config/user/name
    例如，对于`Li`，应该是`https://example.com/config/user/Li`,Li自身和admin有管理权限
    对于`admin`（全局管理员），应该是`https://example.com/config/user/admin`,只有其自身（或者是所有全局管理员？）有管理权限
- 全局管理员对其他用户的管理权限会多，比如分配密钥组，设置该用户（不）可见不可改不可覆盖的元数据规则？

### /home

预计的链接形式：https://example.com/home/user/test/demo.txt

- 可以在用户管理界面将多个实际挂载地址（之前/raw下的路径结构）作为用户态的根目录。
    将`/`下作为用户`/`挂载，该用户将会拥有所有访问权限。
    将`/Baidu/User/test`作为用户`/Baidu`挂载，在该用户的`/Baidu`目录下将会拥有对`/Baidu/User/test`下的访问权限。
- 公共目录为`/home/guset`，可以关闭。
- 可以添加指向其他用户home目录的快捷方式（仅仅只是快捷方式，权限验证应该由前端状态保存）。
- 可以指定每个目录下的权限（读、写、仅列出等），需要考虑冲突：
    1、高权限覆盖低权限
    2、按先后顺序覆盖
- webdav、ftp等应该与权限配置相同，只保留开关。

### 访问密码

- 最大的问题是和webdav和ftp等冲突，是否有必要引入二次验证

基于用户路径添加，在当前用户的状态下访问时需要输入额外的密码。

### 加解密配置（后端）

- 在后端处理，密钥组保存在后端。
- 基于用户路径显示，按真实路径添加，可以配置指定密钥组，任何具有访问该路径权限的人均可解密（加密工作在文件层，无用户区别）。

### 加解密配置（前端）

- 基于网页前端进行解密，只有特定的加密形式支持
- 基于用户路径显示，按真实路径添加。只有该用户可以访问。前端的加解密路径保存在后端，前端丢失时提醒。
- 对于非网页前端访问，按原样提供加密后的文件或者

### 元数据

元数据由写入原始路径的存储文件层，由用户管理负责双向翻译并隐藏真实路径。

### 用户目录负载均衡
用户目录负载均衡这个要考虑一下：
例如/path1/和/path2/作为用户目录，有一个用户user1访问sub路径

1、无状态的，每次访问某个目录，都需要一起访问所有负载挂载目录
那么每次访问/sub都需要访问/path1/user1/sub和/path2/user1/sub


2、有状态的，需要额外维护一个虚拟目录，记录每个文件的实际路径
需要有个表记录sub路径存储于/path1/还是/path2/，设计上会很复杂
